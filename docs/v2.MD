# Cashregister v2 BYOD Architecture

> This document describes the intended v2 architecture for Bring Your Own Device order taking and remote printer spooling. It is a design document for a future release, not an implementation plan and not a description of the currently implemented v1 system.

## Purpose

Cashregister v2 is a Bring Your Own Device release. Cashiers use devices they already own, such as tablets, notebooks, and phones, to open the Cashregister web application over the internet, authenticate, and take orders.

BYOD cashier devices must not connect directly to receipt printers. They may run different operating systems, browsers, USB stacks, network policies, and permission models. The browser remains an order-taking client only.

The core v2 printing problem is routing receipt work from authenticated browser sessions to physical ESC/POS printers without relying on browser-local printer access. The API owns durable print jobs, and a small spooler daemon attached to each printer fetches and executes those jobs.

## Core Decisions

- `PrintStation` is the routing concept for printing. Jobs are not bound directly to cashier devices.
- One spooler daemon is assigned to one print station.
- The normal installation shape is `n` cashiers using `n` print stations and `n` printers.
- Severely constrained installations may route multiple cashiers to one shared print station.
- The API database is the source of truth for orders, print stations, daemon identities, daemon assignment, print jobs, and print-job payloads.
- Daemon state and queueing are ephemeral. The daemon acts on API state; it does not own durable truth.
- Printing is at-least-once. Duplicate receipts are acceptable.
- A daemon completes a job when it has written the ESC/POS bytes to a local file device such as `/dev/usb/lp0`. This means "written", not proven physically printed.
- Receipts must show the order number so duplicate prints of the same order are recognizable.
- One user belongs to one tenant. Users do not switch tenants inside Cashregister.
- Daemons authenticate with machine-to-machine credentials from the identity provider by default. Mutual TLS is an optional stronger transport-level identity model, not the baseline v2 requirement.

## Components

The BYOD web client is the authenticated cashier interface. It runs in a browser on any internet-connected device and talks only to the Cashregister API.

The identity provider is Zitadel or an equivalent cloud OIDC provider. It authenticates users, provides the stable user identity and tenant or organization claim used by the API, and issues machine-to-machine credentials for daemons. Tenant selection is not a UI workflow because each user belongs to exactly one tenant.

The API server owns authorization, order creation, print station selection, print job creation, daemon leasing, retry policy, reprint behavior, and observability. It exposes normal user-facing APIs and a separate daemon-facing API surface.

The database is the durable source of truth for tenants, users, cashier session state, print stations, daemon identities, daemon credential metadata, daemon assignment, print jobs, print-job payloads, and orders. It must not be the source of truth for identity-provider secrets.

The spooler daemon is a small .NET worker running on the Linux device attached to the physical printer. It initiates outbound communication to the API, leases jobs for its assigned print station, writes ESC/POS output to the local printer file device, and acknowledges job results.

The ESC/POS printer remains a local file-device target from the daemon's perspective. The target path is expected to be a writable Linux printer device such as `/dev/usb/lp0`.

The observability target is the Grafana LGTM stack: Loki for logs, Grafana for dashboards, Tempo for traces, and Mimir or a Prometheus-compatible backend for metrics.

## Data Model Concepts

`Tenant` represents one operational customer or installation boundary. Users, print stations, orders, and print jobs are scoped to a tenant. Daemon identities are tenant-neutral until assigned; their effective tenant authority comes only through the Cashregister print-station assignment.

`User` represents an authenticated cashier or admin. The user's tenant comes from identity-provider claims and is not selected in the Cashregister UI.

`CashierSession` represents the active cashier context. It records or resolves the selected print station for order placement. In stateless browser deployments, the browser may store the selected station locally, but the API must validate that station against the user's tenant on every order.

`PrintStation` is a logical printer destination such as "Register 1", "Counter", or "Kitchen". Orders create print jobs for a print station. Multiple cashiers may share a station in constrained installations, but one station serializes through one daemon and one physical printer.

`Daemon` represents a spooler process known to the API. A daemon can be unassigned, assigned to at most one print station, or revoked. A print station can have at most one active assigned daemon.

`DaemonCredential` represents credential metadata for a daemon identity. In the default v2 model the actual credential is a Zitadel machine-to-machine service account or client credential, preferably authenticated with a private-key JWT rather than a shared client secret. Cashregister may store identity-provider ids, key ids, status, and timestamps, but it should not store daemon private keys or long-lived shared secrets. Credentials identify daemons, not tenants. Tenant and print station assignment stays in the database so assignment can change without issuing new daemon credentials.

`PrintJob` is the durable API-owned unit of work for printing. It references the tenant, order, print station, status, immutable print payload, lease metadata, retry metadata, and timing information. The implementation plan may choose whether the payload is stored as a portable `PrintProgram`, encoded ESC/POS bytes, or another versioned representation, but the daemon must not reconstruct receipt business content from mutable order state.

`Order` remains the business record for what the cashier sold. Order placement creates both the order and the initial print job in one API transaction.

These are model concepts, not final table names or DTO contracts. Exact schema and endpoint shapes belong in the implementation plan for v2.

## Print Flow

1. The cashier opens the web application and logs in through the identity provider.
2. The API derives the tenant from the user's claims.
3. The cashier selects a print station or receives a previously configured/default station.
4. The cashier places an order.
5. The API creates the order and its initial print job, including the print payload, in the same transaction.
6. The daemon assigned to the selected print station leases pending jobs from the daemon API.
7. The daemon receives the print payload, encodes it as ESC/POS bytes if needed, writes those bytes to `/dev/usb/lp*`, and acknowledges success or failure.
8. Reprint creates a new print job for the same order. It does not mutate the previous job.

## Daemon/API Communication

Daemon/API communication is daemon-initiated. The daemon makes outbound requests to the API so deployments do not depend on LAN discovery, inbound daemon ports, browser-to-daemon calls, WebRTC, MQTT, NATS, RabbitMQ, or other message-broker infrastructure in v2.

The v2 default is long polling. The daemon keeps a hanging lease request open for its assigned print station. When a job becomes available, the API returns it immediately. When no job becomes available before the request timeout, the daemon repeats the request with normal backoff and jitter.

Leasing is explicit. Pending jobs become leased to a daemon with a lease token and lease expiration. The daemon must include the lease token when acknowledging completion or failure. If the daemon crashes or loses connectivity and the lease expires, the API may return the job to pending for another attempt by the same assigned station daemon.

Every authenticated daemon request should update daemon last-seen metadata. Online and offline status should be derived from recent daemon activity, primarily long-poll lease requests, rather than from a separate inbound health check.

Write failures become failed jobs with diagnostic details and retry metadata. If the daemon writes successfully but loses connectivity before acknowledging, the API may retry the job after lease expiry. This can produce a duplicate receipt and is an accepted consequence of at-least-once printing.

## Daemon Identity and Pairing

Daemon ingress uses ordinary HTTPS and application-layer machine-to-machine authentication by default. Each daemon has its own identity-provider service account or OAuth client. Daemons must not share one global client credential.

The preferred daemon authentication method is private-key JWT or an equivalent asymmetric machine-to-machine flow. Shared client secrets are acceptable only as a temporary bootstrap mechanism because they are harder to protect and rotate on small physical devices.

The daemon identity is derived from validated token claims, such as the token subject or client id. The API maps that stable identity to a `Daemon` record, then authorizes leases through Cashregister-owned tenant and print station assignment. The daemon token alone must not grant tenant authority.

Daemon credentials are tenant-neutral. The API stores daemon identity and credential metadata separately from tenant and print station assignment. This allows a daemon to be reassigned without issuing new identity-provider credentials.

Pairing works as follows:

1. A daemon starts with its own machine-to-machine credential.
2. The daemon obtains an access token from the identity provider.
3. The daemon calls the Cashregister daemon API over HTTPS with the bearer token.
4. The API validates issuer, audience, token lifetime, required daemon scope, and daemon identity claim.
5. The API records or updates the daemon as authenticated and unassigned if it has no print station.
6. The admin UI shows authenticated unassigned daemons as available.
7. An admin assigns the daemon to a tenant print station or creates a print station during assignment.
8. The daemon can then lease jobs only for its assigned print station.

Revocation must prevent future daemon leases. Revocation can happen by disabling the daemon in Cashregister, disabling the identity-provider service account or client, or rotating/removing the daemon's key. A disabled daemon may still hold an unexpired token, so daemon access tokens should be short-lived.

Mutual TLS remains a valid alternative for deployments that need transport-level device identity. In that model, the daemon identity should live in a URI Subject Alternative Name, for example `urn:cashregister:daemon:<ulid>` or `spiffe://cashregister/daemon/<ulid>`. Certificate Common Name may contain a human-readable label, but authorization must not depend on CN.

## Ingress and TLS

HTTPS and mTLS are one TLS handshake, not two sequential protocols. Normal HTTPS authenticates the server to the client. HTTPS with mTLS authenticates both the server and the client during the same TLS handshake.

Normal user traffic and daemon traffic should use separate ingress hostnames:

- `app.cashregister.example.com` serves the web application and user-facing API over ordinary HTTPS.
- `daemon.cashregister.example.com` serves the daemon API over ordinary HTTPS and requires valid daemon machine-to-machine bearer tokens.

With the default machine-to-machine model, Scaleway may terminate public HTTPS and forward traffic to the backend through the existing private path. The daemon API remains protected by token validation in the API. This avoids TLS passthrough and avoids running a Cashregister certificate authority as a v2 requirement.

The daemon API must validate issuer, audience, expiration, and required daemon scope on every request. It must map the token subject or client id to a known daemon and then enforce the Cashregister database assignment before leasing jobs.

Mutual TLS is an optional alternate ingress model. If mTLS is used, the recommended daemon ingress is Scaleway TCP/SSL passthrough to Caddy. Scaleway forwards the encrypted TCP connection without terminating TLS. Caddy terminates TLS, presents the public server certificate, requests and verifies the daemon client certificate, strips any inbound daemon identity headers, and forwards verified daemon identity to the API over a trusted private hop.

Caddy cannot enforce daemon mTLS if Scaleway terminates TLS and forwards plain HTTP. In that topology, the TLS handshake happens at Scaleway and Caddy never sees the daemon client certificate. This limitation does not apply to the default machine-to-machine model because daemon identity is validated from bearer tokens at the API layer.

SSL bridging does not authenticate the original daemon to Caddy. It creates one TLS connection from daemon to load balancer and another from load balancer to Caddy. Caddy can authenticate the load balancer in that shape, but not the original daemon, unless the load balancer itself verifies client certificates and forwards verified identity through a documented and protected mechanism.

## Daemon Network Onboarding

Daemon Wi-Fi configuration must not require `adb`, direct shell access, `nmcli` knowledge, or a Linux workstation during normal operation. Those tools are acceptable only for development, manufacturing, or break-glass support.

The daemon should support two network modes:

- normal mode, where the daemon connects to a configured Wi-Fi network and exposes a small local status page on that LAN;
- setup mode, where the daemon starts its own temporary Wi-Fi access point and captive-portal-style HTTP setup page.

Setup mode is entered when the daemon has no saved Wi-Fi profile, when first boot has not completed network setup, or when all saved Wi-Fi connection attempts fail. The baseline failure threshold is five connection attempts of twelve seconds each, for a total of about sixty seconds. After that threshold, the daemon stops trying to join the unavailable network and enters setup mode.

Setup mode is not a factory reset. It is a temporary network-provisioning state. Existing daemon identity, print-station assignment, and non-network configuration should remain intact unless the operator explicitly requests a destructive reset.

The setup access point should use a per-device SSID such as `Cashregister-Setup-<short-id>` and a per-device setup password. The setup page should let the operator select or enter the target Wi-Fi network, enter the password, and apply it. If the hardware cannot run access-point and Wi-Fi client mode at the same time, applying a new Wi-Fi profile should be treated as a candidate commit: save the candidate profile, stop the setup access point, try the candidate network, and automatically return to setup mode if the candidate does not connect within the configured threshold.

The daemon should expose a compact local status page in normal mode. At minimum it should show daemon id, daemon version, current network name, local IP address, API connectivity state, assigned print station when known, printer target path, and last relevant error. This page is a local operational aid, not a general device-management platform.

The setup access point should be time-limited when possible and should avoid a shared global password. If a future hardware enclosure allows it, a dedicated setup/reset button is preferred: a short press can print status, a long press can enter setup mode, and a very long press can forget Wi-Fi profiles. That button is not a v2 baseline requirement because it requires explicit communication between the Arduino microcontroller and the Linux CPU.

Bluetooth provisioning is optional future work. It must not use an unauthenticated "pair with anything" mode for configuration. Captive-portal setup over a daemon-created Wi-Fi access point remains the baseline onboarding path.

## Failure Modes

If the daemon is offline, jobs remain pending or leased until their leases expire. The cashier UI can show the selected print station as offline or degraded based on daemon last-seen age. Operators can reprint later after the daemon returns.

If the daemon cannot reach the API, it cannot lease new work or acknowledge completed work. The API remains the source of truth. Already written but unacknowledged jobs may be retried after lease expiry.

If the printer is unavailable or `/dev/usb/lp*` cannot be opened or written, the daemon reports the job as failed with diagnostic details. The admin UI and cashier UI should make failed print status visible, and the cashier can trigger a reprint after the operator resolves the printer problem.

If the daemon writes successfully but fails to acknowledge, the API may retry the job after the lease expires. This may print a duplicate receipt. The receipt order number makes the duplicate recognizable.

If a lease expires, the API may return the job to pending. Lease expiry is recovery behavior, not proof that the first attempt did or did not reach the printer.

If multiple cashiers share one print station, the station queue serializes jobs. Operators should expect receipts from different cashiers to interleave according to server queue order.

If the browser loses connectivity after submitting an order, the API result determines behavior. If the order transaction committed, the print job exists and the daemon can still print it. The UI should recover by refetching order state rather than creating a blind duplicate order.

## Latency and Performance

The daemon must use hanging long-poll requests, not fixed multi-second polling. A fixed five-second polling loop would make printing feel unreliable even when the system is healthy.

The v2 latency target is sub-second typical time from committed order to daemon write start on stable connectivity. The system should measure actual queue and write timings before introducing lower-level transports such as WebSockets or message brokers.

Printing is serialized per physical printer. A daemon writes one job at a time to its local file device.

Daemon prefetch should stay low in v2, normally one leased job at a time. Low prefetch keeps recovery behavior easier to reason about when the daemon loses power or connectivity.

## Observability

Cashregister v2 observability is product and workflow observability, not general host monitoring. CPU, RAM, disk, process, and broad device-management telemetry are not the primary goal. The required outcome is that an operator can remotely understand whether orders are being accepted, print jobs are being leased, receipts are being written, daemons are polling, and failures are visible enough to decide what field intervention is needed.

The API server and daemon must both be instrumented with OpenTelemetry for traces, metrics, and structured logs. Application code should use explicit `ActivitySource` and `Meter` instrumentation for Cashregister operations in addition to framework instrumentation. Export should use OTLP so the backend, daemon, local development stack, and Grafana Cloud path share one signal model.

Production telemetry should flow through a deliberate OpenTelemetry Collector or Grafana Alloy gateway before Grafana Cloud. The applications export OTLP to the collector. The collector batches, retries, applies trace tail sampling, and forwards metrics, logs, and traces to Grafana Cloud. Grafana Cloud credentials should live at the collector/gateway boundary, not be copied into every daemon when avoidable.

Trace sampling is collector-side tail sampling by default. The .NET SDK should keep tracing enabled with an always-on sampler so the collector can decide after observing the trace outcome. The collector should always keep error traces and keep only ten percent of successful traces. Expected application failures that represent failed operations, such as printer write failure, daemon authentication failure, lease acknowledgement failure, or invalid print-station assignment, must mark the relevant span with error status so the tail sampler can retain the trace.

The API must emit at least these product metrics:

- orders created;
- print jobs created;
- print jobs leased;
- print jobs written;
- print jobs failed;
- print jobs retried;
- print jobs whose lease expired;
- daemon lease requests by outcome;
- queue depth per print station;
- time from job creation to lease;
- time from lease to written or failed;
- time from order creation to written or failed.

The daemon must emit at least these product metrics:

- poll requests by outcome;
- poll request duration;
- receipts written;
- printer write duration;
- printer write failures;
- acknowledgement failures;
- API connectivity failures;
- setup mode entries;
- Wi-Fi connection failures;
- daemon version;
- selected printer target path.

In this architecture, "receipt written" means ESC/POS bytes were successfully written to the configured printer file device. It does not prove that paper physically printed.

Traces must connect the product workflow across service boundaries. At minimum they must cover order placement, print job creation, daemon long polling, job lease, receipt program construction, local printer write, acknowledgement, reprint, setup-mode entry, and Wi-Fi candidate application. Because printing is asynchronous, daemon print traces may need to link back to order or print-job creation context instead of being a strict child of the original HTTP request.

Logs must be structured and consistently correlated with traces. They must include `tenant.id`, `print_station.id`, `daemon.id`, `print_job.id`, `order.id`, `operation`, `outcome`, and `error.code` whenever those values are known. High-cardinality identifiers should stay as log attributes and trace attributes, not broad Loki labels.

Useful remote alerts should be based on product signals: print job failures, lease expirations, queue depth above threshold, no polling from an assigned daemon during expected service hours, daemon setup mode entered unexpectedly, daemon authentication failures, printer write failures, order created but not leased within a threshold, and order created but not written within a threshold.

If a daemon has no internet path, real-time remote observability cannot work. The daemon may keep bounded local telemetry until connectivity returns, but alerting must treat stale daemon last-seen metadata or a missing poll metric as the remote symptom of the outage.

## Admin UI

The v2 admin UI should stay narrow. It needs only:

- print station list;
- daemon online/offline status;
- daemon last-seen time;
- daemon version;
- daemon last error;
- queue depth per print station;
- pair, assign, revoke, or replace daemon;
- create or rename print stations;
- trigger reprint for failed jobs or existing orders.

The admin UI is not a general device-management platform in v2. Features such as fleet inventory, remote shell, remote package management, and broad hardware diagnostics are out of scope.

## All-In-One Mode

All-in-one deployments must use the same queue and daemon architecture. The daemon may run on the same machine as the API and call the API through localhost, but the API should still create durable print jobs and the daemon should still lease and acknowledge them.

There must not be a separate direct-print path for all-in-one mode. Keeping one print architecture prevents local deployments from drifting away from BYOD deployments.

## Open Decisions

These decisions are not settled by this design document and should be answered before implementation:

- How daemon machine-to-machine credentials are provisioned onto a physical daemon before first network onboarding: manufacturing preload, admin-generated install bundle, local setup upload, or another controlled process.
- Whether the durable print payload stored in `PrintJob` should be a versioned `PrintProgram` or already encoded ESC/POS bytes.
- Whether reprint payloads should be regenerated from current receipt rules or copied from the original order's first print payload.
- Which Zitadel claim is the stable tenant identifier for users, and whether Cashregister stores its own tenant id mapped from a Zitadel organization id.
- Whether all-in-one deployments are still online-first with Zitadel and remote observability, or whether they need a supported offline/local-auth mode.
- Whether constrained installations may intentionally assign multiple physical printers to one operational station in a later release. v2 currently models one active daemon and one physical printer per print station.

## Non-Goals

- Exactly-once printing.
- Browser-local printer integration.
- LAN service discovery.
- Durable daemon spool.
- Multi-tenant user switching.
- Full printer status confirmation.
- Message-broker infrastructure for v2.
- A general-purpose device-management platform.
- A mandatory Cashregister-managed certificate authority for daemon authentication.
